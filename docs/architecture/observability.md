# Наблюдаемость

> Статус: 🟡 design (скелет; реализация — E7.5) · Обновлено: 2026-07-07 ·
> Связанные документы: [deployment](deployment.md), [privacy-mode](privacy-mode.md)

## Принцип: наблюдаемость не должна противоречить приватности

Продукт продаёт локальность данных — поэтому телеметрия проектируется «наоборот»
относительно обычного SaaS:

- **Edge и приватный режим: телеметрия off by default.** Никакие метрики/логи не
  покидают устройство без явного включения владельцем. В приватном режиме — не покидают
  вообще (network-kill это гарантирует).
- Содержимое (аудио, текст стенограмм, имена) **никогда** не попадает в логи и метрики —
  только идентификаторы, длительности, счётчики.
- Cloud-редакция — обычная серверная наблюдаемость, зона ответственности наша.

## Логи

- Структурные JSON-логи (stdlib `logging` + JSON-formatter; в скелете уже `logging`
  вместо `print`).
- Обязательные поля: `ts`, `level`, `component`, `session_id`, `event`.
- Запрещённые поля: текст реплик, имена участников, токены, содержимое кадров.
- На устройстве — journald с ротацией (диск ограничен), уровень по умолчанию INFO.

## Метрики (целевой список)

| Метрика | Тип | Зачем |
|---------|-----|-------|
| `ingest_sessions_active` / `frames_received_total` | gauge/counter | живость приёма |
| `asr_latency_seconds{model}` | histogram | бюджет латентности ([audio-pipeline](audio-pipeline.md)) |
| `llm_latency_seconds{backend=ocp\|local}` | histogram | то же |
| `tts_first_audio_seconds` | histogram | стриминг TTS |
| `e2e_response_seconds` | histogram | критерий EPIC-5: p50 < 3 c |
| `wakeword_triggers_total{result=true\|false}` | counter | FAR/FRR ([ADR-0006](../adr/0006-wake-word.md)) |
| `device_cpu_temp_celsius` | gauge | троттлинг под Whisper |
| `disk_free_bytes{volume=data}` | gauge | переполнение = потеря записи встречи |
| `private_mode_active` | gauge | аудит режима |
| Качество: WER/DER на golden-наборе | offline | считается в CI/бенчах, не в проде |

Экспорт: Prometheus-эндпоинт `/metrics` на ядре (в Edge — доступен только из LAN).

## Health

- `GET /health` — liveness (уже есть в скелете ingest).
- `GET /status` — человекочитаемое состояние для ESP-компаньона и кабинета:
  режим/навык, приватный режим, активная сессия, температура, свободное место.
  Контракт — часть API v1 (EPIC-7).

## Обработка деградаций (что мониторим в первую очередь)

| Сбой | Поведение | Сигнал |
|------|-----------|--------|
| Нет сети (не приватный режим) | очередь на диске + голосовое предупреждение | лог + LED |
| Диск заполнен | стоп записи с явным предупреждением ДО старта встречи | `disk_free_bytes` |
| Перегрев CPU | сброс модели ASR на меньшую, лог | `device_cpu_temp_celsius` |
| Whisper-сервер упал | рестарт (compose `restart`), сессия не теряется (WAV пишется) | health |

## Диагностика парка (пилоты, EPIC-8)

Для пилотных устройств — опциональный «диагностический пакет» по запросу владельца:
архив логов/метрик без содержимого, выгружаемый вручную из кабинета. Постоянный
remote-мониторинг Edge-устройств — только по договору (on-prem/пилот) и никогда —
содержимого.

## Открытые вопросы

- Единый стек для Cloud (Prometheus+Grafana+Loki vs готовый SaaS) — решить в EPIC-7.
- Crash-репорты с устройства: локальный core-dump + ручная отправка?
- Аудит-журнал доступа к записям (кто слушал/читал) — требование 152-ФЗ-контура,
  спроектировать вместе с API v1.
