## <a name="epic-1"></a>EPIC-1. Ядро транскрибации end-to-end (софт, без устройства)

**Цель:** аудиофайл или поток с ПК превращается в русскую стенограмму с сегментами —
на ноутбуке разработчика, `docker compose up`.

**Критерии готовности:** WAV 30-минутной русской встречи → JSON-стенограмма (сегменты,
тайминги) через `core/asr`; сквозной прогон companion-client → ingest → ASR в docker;
WER замерен на тестовом наборе и зафиксирован в docs.

- **E1.1 Вендоринг и запуск Whisper-сервера**
  - [x] Прогнать `vendor_meetily.sh`, зафиксировать `MEETILY_REF` — **v0.4.0** (с v0.4.0 `transcript_processor.py` внутри `backend/app/`)
  - [ ] Собрать `Dockerfile.server-cpu`, скачать модель base (ru) — требует Docker-хоста
  - [ ] Smoke-тест `/inference` вручную + в CI (nightly, не на каждый PR)
- **E1.2 Реализация LocalWhisper (`core/asr/base.py`)**
  - [x] POST WAV(PCM) на `/inference` → парсинг `verbose_json` в `list[Segment]`
  - [x] Обработка ошибок: error-тело сервера, HTTP-статусы, таймаут (ретраи — по мере нужды)
  - [x] Contract-тесты формата сервера Meetily v0.4.0 (`tests/test_asr_localwhisper.py`, MockTransport)
- **E1.3 Связка ingest → ASR**
  - [x] Завершение WS-сессии регистрирует её в SQLite со статусом `queued`
  - [x] Хранение результатов: `core/storage/db.py` (env `VIKAVOICE_DB`; [data-model](architecture/data-model.md))
  - [x] `GET /sessions`, `GET /sessions/{id}/transcript`, `POST /sessions/{id}/transcribe`
  - [ ] Фоновый воркер вместо синхронного POST; квоты/лимиты на сессию (threat model T10)
- **E1.4 Протокол ingest → production-ready (ADR-0009 → accepted)**
  - [x] Аутентификация токеном устройства (`VIKAVOICE_INGEST_TOKEN`, close 1008)
  - [ ] wss:// (терминация TLS реверс-прокси перед ядром); замер трафика тонкого устройства; решение по Opus
- **E1.5 Метрика качества**
  - [ ] Собрать 3–5 тестовых русских записей (публичные + свои, с согласиями)
  - [x] Скрипт подсчёта WER: `core/metrics/wer.py` (S/D/I, нормализация ru); baseline — после записей

## <a name="epic-2"></a>EPIC-2. Диаризация и «Знакомство» (voice enrollment)

**Цель:** стенограмма «от имени»: реплики подписаны именами представившихся участников —
главная BUILD-фича ([reuse-map](compliance/reuse-map.md), §4).

**Критерии готовности:** на записи встречи 3+ участников ≥80% реплик атрибутированы
верно; сценарий «Знакомство» ([scenarios](concept/scenarios.md)) работает через API;
согласие 152-ФЗ фиксируется в данных.

- **E2.1 Диаризация**
  - [x] Интерфейс `core/diarization` + заглушка single в конвейере; DER-метрика (`core/metrics/der.py`)
  - [ ] ADR: pyannote vs tinydiarize (качество/лицензия весов/ресурсы RPi)
  - [ ] Интеграция после ASR: `Segment.speaker = "S1..Sn"`
  - [ ] DER-метрика на тестовом наборе
- **E2.2 Эмбеддер голоса**
  - [ ] Выбор модели (ECAPA-TDNN / SpeechBrain / pyannote-embedding) + проверка лицензии весов
  - [ ] Реализация `embed()` в `core/voice_enrollment/enroll.py`
  - [ ] Валидация порога сходства (текущий 0.62 — заглушка) на реальных голосах
- **E2.3 Сценарий «Знакомство»** — API реализован (`/enroll/*`, `/profiles*`): согласие
  обязательно до приёма голоса, scope meeting|org, безусловный DELETE, зачистка в конце встречи
  - [ ] API: старт enrollment → приём 3–10 c речи → имя на подтверждение
  - [ ] Scope хранения: `meeting_only` (дефолт) vs `org_base` (по согласию)
  - [ ] «Говорящий ?» + переназначение постфактум
- **E2.4 Слой 152-ФЗ для биометрии**
  - [ ] Реализация `CONSENT_RECORD` и retention (`expires_at`) по [data-model](architecture/data-model.md)
  - [ ] Юридическая проработка: 572-ФЗ/ЕБС, уведомление РКН ([открытые вопросы](compliance/152fz.md))
  - [ ] Уведомление о начале записи в API/UX

## <a name="epic-3"></a>EPIC-3. Смысловой слой: протокол, решения, поручения

**Цель:** «после встречи» из сценариев: запись → протокол + решения + action items +
talk-time; локально (Ollama) или через OCPlatform.

**Критерии готовности:** по завершении сессии автоматически формируется структурированный
протокол (MD/JSON); поручения с исполнителем/сроком; экспорт в Telegram и Markdown;
работает полностью офлайн с Ollama.

- **E3.1 Суммаризация** — LLM-абстракция (Ollama/OpenAI-совместимый), русские промпты,
  чанкинг; `POST /sessions/{id}/summarize`
  - [ ] Адаптировать `transcript_processor.py` (Meetily) под наш конвейер
  - [ ] Русские промпты: протокол / решения / поручения
  - [ ] Абстракция LLM-бэкенда (Ollama | OCPlatform | API) — симметрично `ASRBackend`
- **E3.2 Извлечение поручений и аналитика** — ActionItem{assignee,task,due} в схеме
  протокола; talk-time по спикерам (`core/analytics`)
  - [ ] Структурированный вывод `ActionItem{assignee, task, due}` (валидация схемы)
  - [ ] Talk-time по спикерам; обработка меток «важный момент»
- **E3.3 Экспорт** — Markdown-протокол (`GET /sessions/{id}/protocol`); Telegram/Notion — позже
  - [ ] Markdown-протокол
  - [ ] Отправка в Telegram (через OCPlatform)
  - [ ] Заготовки Notion/email — backlog

## <a name="epic-5"></a>EPIC-5. Голосовой ассистент (навык home) на устройстве

**Цель:** разговор с Викой голосом без телефона: wake-word → STT → OCPlatform → Piper →
колонка.

**Критерии готовности:** end-to-end ответ p50 < 3 c (методика замера зафиксирована);
wake-word «Вика» с FAR ≤ 2/сутки, FRR ≤ 10% @ 3 м; та же память, что у Вики в Telegram;
только TLS до gateway.

- **E5.1 Wake-word**
  - [ ] Датасет произношений «Вика» (голоса/дистанции/шум)
  - [ ] Обучение и сравнение openWakeWord vs Sherpa-ONNX; суточный прогон FAR/FRR
  - [ ] Принять [ADR-0006](adr/0006-wake-word.md) по результатам замеров
- **E5.2 Диалоговый цикл (`software/device/`)**
  - [ ] wake → VAD → whisper → OCPlatform (OpenAI-совместимый, TLS, токен из конфига) → Piper → динамик
  - [ ] Barge-in-минимум: прервать TTS по wake-word
  - [ ] LED-состояния: слушаю / думаю / говорю
- **E5.3 Latency-бюджет**
  - [ ] Замеры по стадиям ([бюджет](architecture/audio-pipeline.md#бюджет-латентности-цели-не-замерено)); выбор модели STT
  - [ ] Стриминг TTS по предложениям
- **E5.4 Контракт с OCPlatform**
  - [ ] Спецификация в docs/reference: session ID (задел под multi-device), device-token, TTS-директивы
  - [ ] Требования к OCPlatform из [mobile.md](architecture/mobile.md#что-нужно-от-ocplatform) — согласовать

## <a name="epic-6"></a>EPIC-6. Приватный режим и редакция Edge

**Цель:** «Вика, приватный режим» / Edge-редакция: вся цепочка локально; **доказуемое**
отсутствие трафика ([privacy-mode](architecture/privacy-mode.md)).

**Критерии готовности:** автотест tcpdump «0 пакетов наружу» проходит; встреча с физически
отключённым Ethernet даёт стенограмму и саммари; красная индикация; переключение
голосом/кнопкой/таймером.

- **E6.1 Локальная LLM**
  - [ ] llama.cpp/Ollama на устройстве; бенч на русском: llama-3.2-1B/3B, phi-3.5-mini, qwen2.5-3B (скорость/качество)
  - [ ] Выбор модели с учётом лицензии весов (Llama Community License vs Apache/MIT) — ADR
- **E6.2 Роутер режимов**
  - [ ] Переключение OCPlatformClient ↔ LocalLlamaClient в рантайме
  - [ ] Честная деградация: голосовое сообщение о недоступности памяти/инструментов
  - [ ] Таймер выхода из режима
- **E6.3 Гарантии изоляции**
  - [ ] Network-kill (nftables) + автотест «нет пакетов наружу» (обязателен в CI-on-device)
  - [ ] Профили L2/L3 (строгий без LAN-исключений); физическая airgap-кнопка — в E7.4
- **E6.4 Edge-редакция как артефакт**
  - [ ] Compose-профиль edge полностью офлайн (модели в образе)
  - [ ] Шифрование хранилища записей (LUKS; модель ключей — открытый вопрос threat model)

- **E7.2 Веб-кабинет (BUILD)**

  - [ ] Правка реплик и имён, переназначение «Говорящий ?»

обратной связи; сформулировано ценообразование.
