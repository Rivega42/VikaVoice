# 0009. Протокол приёма аудио (ingest)

- Статус: **proposed** (v1 реализован как скелет; принять после нагрузочной проверки в EPIC-1)
- Дата: 2026-07-07

## Контекст и проблема

В ядро стекается аудио из трёх источников: устройство (микрофонный массив), клиент-компаньон
на ПК (системный звук онлайн-встречи), в будущем — мобильный клиент. Нужен единый протокол:
простой, потоковый, с метаданными источника, пригодный для LAN и интернета.

## Рассмотренные варианты

1. **WebSocket: первый кадр — JSON-заголовок, далее бинарные PCM-кадры** — тривиально
   реализуется на любом клиенте (Python, ESP, браузер), проксируется через обычный HTTPS,
   достаточно для записи встреч (не сверхнизкая латентность).
2. **WebRTC (LiveKit и т.п.)** — лучшая латентность/джиттер, встроенный Opus; но тяжёлая
   инфраструктура (SFU/TURN) — оправдана для realtime-разговора (mobile, EPIC-9), не для
   записи встреч.
3. **gRPC streaming** — типизированный контракт, но неудобно из браузера и с ESP32.
4. **Чистый HTTP chunked upload** — нет обратного канала (подтверждения, команды серверу).

## Решение

Для записи встреч — вариант 1, версия протокола **v1**:

- Эндпоинт: `ws(s)://<core>/ingest`.
- **Первое сообщение — обязательный** текстовый JSON-заголовок:
  `{"v": 1, "rate": 16000, "format": "pcm_s16le", "channels": 1, "source": "system|mic|both|device|test"}`.
  Бинарный кадр до заголовка или повторный заголовок → сервер закрывает соединение с кодом 1003.
- Далее — бинарные кадры PCM (рекомендуемо по 100 мс), моно 16 бит little-endian.
- Полная спецификация — [reference/api/ingest-ws.md](../reference/api/ingest-ws.md).

WebRTC остаётся целевым транспортом для realtime-диалога (EPIC-9) — это дополнение,
а не замена v1.

## Открытые вопросы (блокируют перевод в accepted)

- **Аутентификация**: токен устройства в заголовке (`"token": "..."`) + TLS — обязательны
  до выхода за пределы доверенной LAN (см. [threat model](../architecture/security-threat-model.md)).
- Сжатие (Opus) для тонкого устройства через интернет — замерить трафик.
- Подтверждения/номера кадров при разрывах соединения (resume) — нужно ли для встреч.

## Последствия

- Обе стороны (клиент-компаньон, ядро) уже говорят на v1; изменение формата — только
  с инкрементом `v` и поддержкой предыдущей версии на приёме в течение переходного периода.
